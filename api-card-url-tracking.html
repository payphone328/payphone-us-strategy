Problema Critico

El Problema: Cero Visibilidad en URLs del API Card

Payphone ofrece 3 metodos de integracion de pagos. Solo el API Card opera en un modelo donde no tenemos visibilidad del frontend del comercio.

Metodo Como Funciona Visibilidad URL Riesgo Requiere PCI
Boton por Redireccion Usuario es redirigido a pagina hosted de Payphone 100% Payphone controla la pagina Bajo No
Cajita de Pagos Widget/iframe embebido en pagina del comercio Alta iframe origen verificable Bajo No
API Card Comercio recolecta datos de tarjeta en SU form, envia server-to-server a Payphone 0% Solo vemos IP del servidor Critico Si
El API Card es server-to-server. El comercio captura PAN, CVV y expiracion en su propio formulario y los transmite al endpoint de Payphone. Payphone solo ve la llamada API desde el servidor del comercio, NUNCA ve la URL del frontend donde el tarjetahabiente ingreso sus datos. Esto permite que un comercio registre un negocio legitimo pero despliegue el formulario en sitios no autorizados (casinos, contenido adulto, etc).

Ruta de Integracion: ¿Tienes PCI?

No todos los comercios necesitan (ni deben) usar el API Card. Payphone provee certificacion PCI DSS 4.0 y 3D Secure integrado en sus productos hosted — sin costo adicional ni certificacion propia.

✅

Comercio SIN PCI propio

RECOMENDADO — Payphone provee PCI + 3DS

El comercio NO toca datos de tarjeta. Payphone maneja PCI DSS 4.0, 3D Secure, tokenizacion y Decision Manager. Cero riesgo de compliance para el comercio.

Productos disponibles:

  • Boton de Pago (Redireccion) — Usuario va a pagina hosted de Payphone. Maxima seguridad, cero integracion de frontend.
  • Cajita de Pagos (Widget/iframe) — Formulario embebido en la pagina del comercio. Payphone controla el iframe con datos sensibles.
  • Link de Pagos — URL unica generada por API. Ideal para cobros por WhatsApp, email o redes sociales.
PCI incluido 3D Secure incluido Tokenizacion incluida Sin multas Visa

Comercio CON PCI propio

REQUIERE — Certificacion PCI DSS vigente

El comercio captura datos de tarjeta en su propio formulario. Requiere certificacion PCI DSS vigente, Domain Registry obligatorio, y todas las capas de validacion de este documento.

Requisitos obligatorios:

  • Certificado PCI DSS vigente — URL del certificado verificable
  • 3D Secure habilitado — Obligatorio para todas las transacciones
  • Domain Registry — Registro de todos los dominios autorizados
  • JS Beacon — Script obligatorio en pagina de checkout
  • Licencia de betting si categoria = betting
PCI requerido 3DS requerido Alto escrutinio Solo API Card
Valor diferenciador de Payphone: La mayoria de procesadores en LATAM exigen que el comercio obtenga su propia certificacion PCI (costo: $15K-$50K/año). Payphone ofrece PCI DSS 4.0, 3D Secure y tokenizacion integrados sin costo adicional en Boton de Pago, Cajita de Pagos y Link de Pagos. Esto elimina la barrera de entrada para comercios que quieren aceptar tarjetas de forma segura sin la complejidad y costo de certificarse.
Regla de onboarding: Si un comercio solicita acceso al API Card pero no presenta certificado PCI vigente, el sistema debe automaticamente ofrecer las alternativas hosted (Boton, Cajita, Link de Pagos) en lugar de rechazar la solicitud. Esto convierte una denegacion en una oportunidad de venta y evita que el comercio intente usar API Card sin certificacion real (como el caso Charge Money).

Capa 1

Domain Registry: Registro Obligatorio de Dominios

Ningun API Card se activa sin registro previo de dominios autorizados. El comercio declara TODAS las URLs donde se mostrara el formulario de pago.

Campo Tipo Requerido Descripcion
merchantId string Si ID del comercio en Payphone
domains string[] Si Lista de dominios autorizados (ej: ["tienda.com", "checkout.tienda.com"])
businessCategory string Si MCC code o categoria (retail, betting, services, etc.)
pciCertificateUrl string Si URL del certificado PCI DSS vigente
bettingLicenseId string Condicional Obligatorio si businessCategory = "betting"
webhookUrl string Opcional URL para recibir alertas de compliance

Verificacion de Dominio (DNS TXT)

Similar a Google Search Console. El comercio agrega un registro TXT a su DNS:

// DNS TXT Record
betcris.com  TXT  "payphone-verify=a1b2c3d4e5f6"

Payphone verifica automaticamente. Si el TXT no existe o se remueve, el dominio se desactiva.

Reglas del Registry

  • Maximo 10 dominios por comercio (ampliable con aprobacion)
  • Subdominios wildcard NO permitidos (*.tienda.com)
  • Cada dominio requiere verificacion DNS independiente
  • Re-verificacion automatica cada 30 dias
  • Agregar nuevos dominios requiere aprobacion manual si categoria = betting
Capas 2 y 3

JS Beacon + API Middleware

Doble validacion: el beacon captura la URL real del frontend, y el middleware la valida contra el Domain Registry antes de procesar.

Capa 2: JavaScript Beacon

Script obligatorio que el comercio debe incluir en la pagina donde se muestra el formulario de pago. Captura la URL real del browser y genera un beaconSessionId.

payphone-beacon.js — Script del comercio 
// El comercio incluye esto en su pagina de checkout
<script
  src="https://cdn.payphone.app/beacon/v1/beacon.js"
  data-merchant="MCH-2025-00142"
  data-store="STORE-001"
></script>

// El beacon automaticamente:
// 1. Captura window.location.href
// 2. Captura document.referrer
// 3. Genera fingerprint del DOM
// 4. Detecta si esta en iframe
// 5. Envia POST a Payphone
// 6. Retorna beaconSessionId

// El comercio obtiene el ID:
const beaconId = await PayphoneBeacon.getSessionId();
// Debe incluirlo en el request al API Card

Capa 3: API Middleware

Intercepta cada llamada al API Card y ejecuta 4 validaciones antes de permitir el procesamiento.

Logica del Middleware (pseudocodigo) 
function validateCardRequest(req) {
  // 1. Extraer beaconSessionId del request
  const beacon = getBeaconSession(req.beaconSessionId);

  // 2. Verificar que beacon existe y no expiro
  if (!beacon || beacon.expired)
    return { "error": 901, "msg": "Beacon invalido" };

  // 3. Extraer dominio del URL capturado
  const domain = extractDomain(beacon.capturedUrl);

  // 4. Verificar contra Domain Registry
  const registry = getDomainRegistry(req.merchantId);
  if (!registry.domains.includes(domain))
    return { "error": 902, "msg": "Dominio no autorizado" };

  // 5. Verificar que no hay iframe embebido
  if (beacon.isInIframe)
    return { "error": 903, "msg": "Iframe no permitido" };

  // 6. Log para auditoria
  logTransaction(req.merchantId, domain, beacon);

  return { "status": "approved" };
}
Que pasa si el comercio no incluye el beacon? El API Card rechaza la transaccion con error 901 - Beacon session required. Esto es un cambio breaking para comercios existentes, por lo que se implementa con periodo de migracion (ver Roadmap).

Que Captura el Beacon

Dato Metodo de Captura Proposito Evasion Difficulty
URL completa window.location.href URL real donde se muestra el form Dificil de falsificar
Referrer document.referrer Detectar redirecciones Media
Es iframe window !== window.top Detectar embedding no autorizado Dificil
DOM fingerprint Hash de elementos del DOM Detectar paginas clonadas Dificil
TLS certificate Verificacion server-side Confirmar identidad del dominio Muy dificil
Timestamp + nonce Generado por beacon Prevenir replay attacks Imposible
Limitacion conocida: Un comercio tecnico sofisticado podria correr el beacon en un server headless (Puppeteer) y falsificar la URL. Para mitigar esto, la Capa 4 (URL Scanner) verifica independientemente que el dominio registrado realmente tiene un formulario de pago Payphone.
Capa 4

URL Scanner: Verificacion Asincrona Continua

Servicio de crawling automatizado que visita los dominios registrados periodicamente para verificar que efectivamente muestran un formulario de pago legitimo de Payphone.

Scan de Contenido

Visita cada URL registrada y analiza:

  • Presencia del beacon script de Payphone
  • Tipo de negocio real (ecommerce, casino, adulto)
  • Coincidencia con MCC declarado
  • Presencia de campos de tarjeta

Scan de Redirecciones

Detecta si el dominio registrado redirige a otros sitios:

  • HTTP 301/302 redirects
  • JavaScript redirects (window.location)
  • Meta refresh redirects
  • iframe loading de otros dominios

Deteccion de Categorias

Clasifica automaticamente el contenido del sitio:

  • Gambling / Casino keywords
  • Adult content indicators
  • Pharma / weapons keywords
  • Comparacion vs MCC registrado
Logica del Scanner — Reglas de Alerta 
// Frecuencia de scans
const SCAN_FREQUENCY = {

Caso Abarth Solution con este sistema: El scanner habria detectado las 2 URLs de casino que no estaban en el Domain Registry del comercio. Ademas, el beacon habria reportado URLs no registradas en tiempo real. Doble capa de proteccion.

Compliance

Alineacion con PCI DSS 4.0

Esta solucion no solo resuelve el problema de visibilidad — tambien nos alinea con los nuevos requisitos de PCI DSS 4.0 que seran obligatorios.

Requisito PCI DSS 4.0 Que Exige Como lo Resuelve Nuestra Solucion Capa
6.4.3 Monitoreo de integridad de scripts en paginas de pago. Todos los scripts en la pagina deben estar autorizados y verificados. El JS Beacon valida que solo scripts autorizados estan en el DOM. Reporta cualquier script de terceros no reconocido. L2 Beacon
11.6.1 Deteccion de cambios no autorizados en paginas de pago. Alertas cuando el contenido de la pagina cambia. El URL Scanner hace hash del DOM en cada visita. Si el contenido cambia significativamente, se dispara un re-scan inmediato. L4 Scanner

6.3.2 Inventario de software custom y componentes de terceros. El beacon reporta todos los scripts presentes en la pagina de pago, creando un inventario automatico. L2 Beacon

Decisiones Requeridas

Puntos de Decision para Val + Juan Diego

Decisiones que necesitan aprobacion antes de iniciar desarrollo.

URGENTE Decision 1

Suspender nuevas integraciones API Card?

Con 2 multas en 6 meses, se recomienda suspender nuevas activaciones de API Card hasta que el Domain Registry (Fase 1) este en produccion. Solo mantener Betcris activo.

Riesgo de no actuar: Una tercera multa puede resultar en multas incrementales de Visa ($25K-$100K) y posible revocacion de BIN sponsorship.
ESTRATEGICA Decision 2

Beacon obligatorio o solo Domain Registry?

El Domain Registry (L1) + Scanner (L4) resuelve el 80% del problema. El beacon (L2) + middleware (L3) agrega el 20% restante pero requiere cambio en la integracion de todos los comercios.

Opcion A: Solo L1 + L4 (rapido, menor impacto en comercios)
Opcion B: L1 + L2 + L3 + L4 completo (mayor seguridad, breaking change para comercios)
TECNICA Decision 3

Asignacion de equipo de desarrollo

Se necesitan 2 devs backend + 1 dev frontend por ~8 semanas. Juan Diego debe definir quienes del equipo pueden asignarse y confirmar capacidad.

Dependencia: Si hay conflicto con otros proyectos (Pipedrive, Insider, expansiones), priorizar este proyecto por riesgo de compliance.
COMERCIAL Decision 4

API Card como producto premium para betting?

Con estos controles, el API Card se convierte en un producto de alta seguridad ideal para el vertical de betting. Podemos posicionarlo como diferenciador competitivo con pricing premium.

Oportunidad: "API Card Secure" con Domain Registry + Scanner como feature premium para operadores de betting. Fee adicional por monitoreo continuo.
